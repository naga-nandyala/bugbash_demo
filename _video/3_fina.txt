# overview

Change is coming to Azure CLI on macOS. Two new ways to install — and a major authentication upgrade.

First, Homebrew users will see Azure CLI moving from the old formula install to a Homebrew Cask. That means a self-contained, pre-built bundle with native binaries. The new method is designed for reliability and security.

Second, for air-gapped environments and restricted networks, we're introducing architecture-specific offline tarballs. Download from GitHub, extract for your platform — ARM64 or Intel — and point to any compatible Python. No package manager needed. It's flexible, fast, and ready to go where Homebrew can't.

But that's not all. We're bringing seamless broker-based authentication through Company Portal SSO — the same login experience you already have on Windows, now on macOS. When Company Portal is installed, Azure CLI uses the broker instead of the browser. It's that simple.

These changes need to be validated on real hardware, with live Azure deployments. We need to prove every install method and every login flow works end-to-end, in the environments you use every day.

That's what this bug bash is for. Let's walk through how it's structured.


## phases

Six phases, each guided and orchestrated through GitHub Copilot — designed to build confidence that every install method and login flow is production-ready.

Phase one: Existing State. Before changing anything, we document the machine’s status—Python runtimes, Azure CLI, and its extensions. We validate functionality, then perform a clean uninstall to ensure a blank slate.

Phase two: Cask Install. We challenge the new Homebrew Cask installation from end to end—checking signatures, logins, extensions, upgrades, and ensuring a clean uninstall after reinstall.


Phase three: Offline Install. For environments without Homebrew, we download the tarball, inspect security attributes, set up standalone Python, and run the CLI independently.

Phase four: Ring Zero. Installations aren’t enough. We deploy eight core Azure services as an interconnected architecture, verify each, then clean up—testing real-world utility.

Phase five: Broker Authentication. We test the new macOS login via MSAL and Company Portal—verifying broker login, browser fallback, and toggling the flow for validation.

Phase six: Telemetry Verification. Observability ensures installation and authentication flows are fully captured, with each attribute reaching the backend for analysis.
